Anything in the this branch is deployable

This is basically the only hard rule of the system. There is only one branch that has any specific and consistent meaning

This branch is stable and it is always, always safe to deploy from it or create new branches off of it. If you push something to master that is not tested or breaks the build, you break the social contract of the development team and you normally feel pretty bad about it.

Create descriptive branches off of master

Hotfix

When you want to start work on anything, you create a descriptively named branch off of the stable master branch.

This has several advantages - one is that when you fetch, you can see the topics that everyone else has been working on. Another is that if you abandon a branch for a while and go back to it later, it’s fairly easy to remember what it was. It’s almost like a list of upcoming features with current rough status.

Feature

The fact that source branch and target branch are the same (it mean, without any other change was added) is an ideal situation because will not occur conflicts.

We have manage our task to find the way that programming go agead in this scenario. Imagine

This is really important. It is the key for future coding strategy.

Refactoring

It really does not matter how big is the change. You must always be consistent by forking from and merging to the stable branch.

Open a pull request at any time

Many people use it for open update the project, or send a pull request to the maintainer bug You can also send pull requests from one branch to another, so you can use them to say “I need help or review on this” in addition to “Please merge this in”. Be social!

We use the Pull Requests to review the code long before we actually want to merge it into master for deployment. If you are stuck in the progress of your feature or branch and need help or advice, or even if you have little or no code but some screenshot comps or general ideas, you open a pull request.

Push to named branches constantly

Since the only thing we really have to worry about is the stable branch from a deployment standpoint, pushing to the server doesn’t mess anyone up or confuse things - everything that is not inside the stable branch is simply something being worked on. In consecuense, any not-merged branch can be tested, approved, discarded and deleted.

Pushing everything to server your are backupping your work and more importantly, it puts everyone in constant communication. A simple git fetch will basically give you a TODO list of what every is currently working on.

Use your initials letter as a branch prefix. For example. My name is Piero Blunda, and I create a branch named pb/feature-description. Work on one branch a time and focus to merge or discard the changes before opening a new one.

Dos cambios en serie

All changes in the blue branch are included in the orange one. This usually happens when works only one programmer in the project, but also happen when blue branch and orange branch are not related at all.

Again, this is an ideal situation, but it is absolutelly reachable if your team split task correctly. We need to work to being a ninja splitter taks instead of resolving conflicts.

Cambios en paralelo

In this cases, changes in each feature branches are not shared. This is very common situation when multiple programmers works simulateneously.

It would be a big problem if you need a piece of code of the other branch. Its happens when those branches are related, for example, when two programmers are working in the same section of the proyect. It generates conflicts, or generates unredeabble git-tree.

It absolutelly will not represent any problem if you are splitting task in the right way and you are a non related task inside the same operation.

Avoid pulling commits from other branches. That branches are not stables, so you can not merge it into your branch. There are only one stable branch, isn't it?

If you need the code in the blue branch, that task must be developed serially in different iterations.

Like the example before, void puling from any not stable branch. In this cases, both branches are dependent from each other. It means that they are really the same task. It is ok if you can not split a task. Manage the large branches correctly.

Conflict

The best solving-conflict strategy is do not to need to resolve them. Splitting task correctly and planning the Sprint smarty results in less conflicts to resolve, so, more time for programming important thinks.

In the rare situation you need to resolve a conflict, you only need to update your working branch from the stable branch in order to set up to date respect the target-branch.
