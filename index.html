The stable branch

gold rule Anything in the this branch is deployable. There is only one branch that has any specific and consistent meaning.

This branch is stable and it is always , always safe to deploy from it or create new branches off of it. If you push something to this branch that is not tested or breaks the build, you break the social contract of the development team and you normally feel pretty bad about it.

Create descriptive branches off of the stable branch

gold rule When you want to start work on anything, you create a descriptively named branch off of the stable branch.

Hotfix

This has several advantages - one is that when you fetch, you can see the topics that everyone else has been working on. Another is that if you abandon a branch for a while and go back to it later, it’s fairly easy to remember what it was. It’s almost like a list of upcoming features with current rough status.

Feature

The fact that source branch and target branch are the same (it mean, without any other change was added) is an ideal situation because will not occur conflicts.

We have manage our task to find the way that programming go agead in this scenario. Imagine all the people working without conflicts. It's easy if you try. You may say that I'm a dreamer, I'm not the only one. I hope someday you'll join us.

This is really important. It is the key for future coding strategy.

Open a pull request
at any time

Many people use it for open update the project, or send a pull request to the maintainer bug You can also send pull requests from one branch to another, so you can use them to say “I need help or review on this” in addition to “Please merge this in”. Be social!

We use the Pull Requests to review the code long before we actually want to merge it into master for deployment. If you are stuck in the progress of your feature or branch and need help or advice, or even if you have little or no code but some screenshot comps or general ideas, you open a pull request.

Merge only after pull request review

we try to get signoff from someone else in the company. This is generally a +1 or emoji or “:shipit:” comment, but we try to get someone else to look at it.

Test it, and run automated test

Deploy immediately after review

Finally, your work is done and merged into the master branch. This means that even if you don’t deploy it now, people will base new work off of it and the next deploy, which will likely happen in a few hours, will push it out. So since you really don’t want someone else to push something that you wrote that breaks things, people tend to make sure that it really is stable when it’s merged and people also tend to push their own changes.

Pro Tip: tag a commit with a software version to identify the exactly production commit. It is not necessary deploy inmediattely, but you shour go online as soon as possible.

The process is simple, straightforward, scalable and powerful. Issues are addressed quickly if they ever come up at all, and new features are introduced at a rapid pace. There is no compromise of quality or stability so that we can get more speed or simplicity or less process.

Push to named branches constantly

Since the only thing we really have to worry about is the stable branch from a deployment standpoint, pushing to the server doesn’t mess anyone up or confuse things - everything that is not inside the stable branch is simply something being worked on. In consecuense, any not-merged branch can be tested, approved, discarded and deleted.

Pushing everything to server your are backupping your work and more importantly, it puts everyone in constant communication. A simple git fetch will basically give you a TODO list of what every is currently working on.

Use your initials letter as a branch prefix. For example. My name is Piero Blunda, and I create a branch named pb/feature-description. Work on one branch a time and focus to merge or discard the changes before opening a new one.
